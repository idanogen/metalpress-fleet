# heyy Flow Architecture — מצב סופי

**עודכן:** 2026-05-19

## העיקרון

heyy = שלד תזרים. הלוגיקה והאמת — בשרת.

- **heyy** שולח את התבנית, מקבל את התגובה, ומעביר ל-webhook
- **webhook (Vercel)** עושה את **כל** הוולידציה ועדכוני המערכות (Supabase, Priority, kilo ב-heyy)
- **webhook** גם שולח את התשובה לנהג ישירות דרך heyy API
- **AI Employee** — לא בשימוש (הוסר מהתזרים)

## תזרים מומלץ ב-heyy

```
┌─────────────────────────────────────────┐
│ [שלח תבנית — הודעה בתחילת חודש]         │
│   • תבנית: הודעה בתחילת חודש            │
│   • driver_name → {{contact.full_name}} │
│   • שמור תגובת טקסט ל:                  │
│       ✅ משתנה אוטומציה: rawUserReply   │
│       ❌ תכונת איש קשר: (ריק!)          │  ← זה מה ששינינו
└─────────────────────────────────────────┘
                  ↓ הנהג השיב
┌─────────────────────────────────────────┐
│ [קריאת API — POST]                      │
│   URL:    https://metalpress-fleet.vercel.app/api/heyy-webhook
│   Method: POST                          │
│   Body:   {                             │
│     "from": "{{contact.phoneNumber}}",  │
│     "text": "{{rawUserReply}}"          │
│   }                                     │
└─────────────────────────────────────────┘
                  ↓ ה-webhook מטפל בהכל
                  • מאמת מספר
                  • שומר ב-monthly_reports
                  • מעדכן vehicles.current_mileage
                  • מעדכן contact.kilo ב-heyy
                  • שולח ל-Make → Priority
                  • שולח WhatsApp confirmation לנהג

(אופציונלי) — סוף תזרים. אין עוד שלבים.
```

## למה אין יותר AI Employee?

1. **הוולידציה דטרמיניסטית** — חוקי הוולידציה (1 ≤ N ≤ 9,999,999, N > current_kilo, פער ≤ 5000) רצים בקוד, לא ב-LLM. אין מקום לטעויות.
2. **חיסכון** — אין קריאות LLM על כל הודעה.
3. **מקור אמת אחד** — ההודעה לנהג נשלחת מ-`api/heyy-webhook.ts`, לא ממקום אחר.
4. **קל לתחזוקה** — שינוי בתשובות = git commit, לא ניסוי-וטעייה בפרומפט.

## תשובות שה-webhook שולח לנהג

| מצב | טקסט |
|---|---|
| הצלחה | `שלום <שם> 🙏 קיבלנו את דיווח הקילומטרז' שלך לחודש <חודש> <שנה>: <מספר> ק"מ ✅ תודה ויום טוב.` |
| אין מספר בקלט | `<שם>, לא הצלחתי לזהות מספר קילומטראז' תקין בהודעה. נא לשלוח רק את המספר, לדוגמה: 125430` |
| מספר נמוך/שווה לקריאה האחרונה | `<שם>, המספר ששלחת (<N>) נמוך או שווה לקריאה האחרונה (<current>). נא לבדוק ולשלוח שוב.` |
| קפיצה גדולה (>5000 ק"מ) | `<שם>, הקריאה (<N>) גבוהה משמעותית מהקודמת (<current>). נא לוודא שלא נוספה ספרה בטעות ולשלוח שוב.` |
| נהג לא נמצא | _(לא נשלחת הודעה — לא מזוהה)_ |

## הגדרת ה-API call ב-heyy

ב-heyy, ה-node "קריאת API" / "HTTP request" / "Webhook":

| שדה | ערך |
|---|---|
| Method | POST |
| URL | `https://metalpress-fleet.vercel.app/api/heyy-webhook` |
| Headers | `Content-Type: application/json` |
| Body (JSON) | `{"from": "{{contact.phoneNumber}}", "text": "{{rawUserReply}}"}` |
| Timeout | 30 שניות (בגלל הקריאה ל-Make + Priority) |

**אופציונלי:** אם רוצים לאבטח, להוסיף ל-Vercel env `HEYY_WEBHOOK_SECRET` ולשלוח מ-heyy header `X-Heyy-Secret: <אותו ערך>`.

## מה לעשות עם ה-AI Employee הקיים?

1. **השאר אותו** ב-heyy אבל **אל תחבר אותו לתזרים החדש** — אפשר שיישאר לשימוש עתידי.
2. **או** למחוק לגמרי ולנקות.

ההמלצה: להשאיר חבוי, כדי שלא נצטרך לבנות אותו מחדש אם יום אחד נצטרך שיחה רב-שלבית.

## בדיקות אחרי המעבר

1. **רכב הבדיקה (פלטה 11223466)** — kilo=100 ב-heyy ובסופאבייס. שולחים "50" מ-WhatsApp:
   - צפי: הנהג מקבל **400** מה-webhook (לא הודעת WhatsApp) → ה-AI לא מגיב כי הוא לא בתזרים → הנהג לא יקבל כלום ⚠️
2. שולחים "150":
   - צפי: ה-webhook מחזיר 200, kilo הופך ל-150 ב-heyy ובסופאבייס, הנהג מקבל "קיבלנו..."
3. שולחים "היי":
   - צפי: ה-webhook מחזיר 400 — הנהג לא מקבל כלום ⚠️
