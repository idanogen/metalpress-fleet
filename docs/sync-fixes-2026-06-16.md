## 2. תיקון באג "החלטות נדבקות" (migration `sync_divergence_sticky_decisions`)

### שורש הבעיה
ב-RPC `public.sync_vehicle_from_priority`, התנאי שמונע פתיחת סתירה כפולה בדק רק אם קיים
פריט שעדיין `pending_review`. ברגע שהמנהלת טיפלה בפריט (`approved`/`rejected`/`dismissed`)
— בסנכרון הבא נפתח פריט **חדש** עם אותו ערך בדיוק. ההחלטה לא נדבקה.

הוכחה בדאטה: `inbound_messages` ids 175–180 ו-185–190 היו אותן 6 סתירות עם `priority_mileage` זהה.

### התיקון
כל פריט שומר ב-`raw_payload` את `priority_mileage` (ערך הפריוריטי בזמן הסימון). התנאי שונה כך:

```sql
AND NOT EXISTS (
  SELECT 1 FROM fleet.inbound_messages
  WHERE matched_vehicle_id = v_id
    AND parsed_year = v_year AND parsed_month = v_month
    AND provider = 'priority_sync'
    AND (
      status = 'pending_review'                                              -- עדיין פתוח → לא לכפול
      OR coalesce((raw_payload->>'priority_mileage')::int, -1) = coalesce(v_mileage, 0)  -- כבר הוחלט על ערך הפריוריטי הזה → שקט
    )
)
```

### התנהגות
| תרחיש | אחרי התיקון |
|--------|--------------|
| הוחלט, פריוריטי לא השתנה | שקט ✅ |
| פריוריטי השתנה לערך חדש | נפתח שוב ✅ |
| סתירה ראשונה (טרם הוחלט) | נפתח ✅ |
| פריט עדיין פתוח | לא כופל ✅ |

ללא שינוי בפרונט, ב-`/api/resolve-review` או בסנריו Make. עובד רטרואקטיבית (הפריטים הקיימים
כבר שומרים `priority_mileage`).
